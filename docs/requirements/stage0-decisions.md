# Решения этапа 0: оплата и печать

Зафиксированные выборы для реализации (можно уточнить перед стартом разработки).

---

## 1. Оплата

### Модель (простыми словами)

- Гость нажимает «Оплатить» в нашем сервисе → наш бэкенд создаёт платёж у платёжной системы и отдаёт ссылку на оплату (или открывает окно оплаты). Гость переходит по ссылке, платит картой/СБП на стороне банка или платёжки.
- Наш сервис **узнаёт об успешной оплате не от браузера гостя**, а по специальному уведомлению (webhook), которое платёжная система шлёт нашему серверу. Так мы не зависим от того, закрыл ли гость вкладку, и не можем «обмануться» двойным нажатием: один заказ — один платёжный запрос, повторное нажатие «Оплатить» для того же заказа просто вернёт ту же ссылку.

### Провайдер оплаты

**Зафиксировано для реализации сейчас**:

- **Приём денег**: **Точка.API → платёжные ссылки** (гость платит по ссылке на стороне банка).
- **Фискализация чеков (54‑ФЗ)**: **через Точку** (создание ссылки **с чеком** / `With Receipt`, где мы передаём позиции и контакт гостя для отправки электронного чека).
- **План Б**: быть готовыми фискализировать **через Эвотор «Цифровая касса»** отдельно (см. ниже), если понадобится другой фискальный контур.

Контекст (важно правильно понимать роли):

- **Точка** — банк/эквайринг: принимает оплату (карта/СБП) и умеет в «платёжные ссылки».
- **Эвотор (Цифровая касса)** — фискальный контур: формирует/регистрирует фискальные чеки и передаёт данные в ОФД/ФНС.

### Ключевые принципы (чтобы не было ошибок в проде)

- **Webhook — единственный источник истины**: редирект гостя на `redirectUrl` не означает, что платёж успешно завершён (гость может закрыть вкладку, может быть задержка процессинга, может быть повтор вебхука).
- **Идемпотентность**:\n  - создание оплаты для одного `order_id` должно возвращать **одну и ту же** ссылку (пока платёж не завершён/не истёк);\n  - обработчик webhook должен безопасно переживать **повторы** (один `operationId`/событие может прийти несколько раз).
- **Связка**: `order_id` ↔ `paymentLinkId` ↔ `operationId` (и всё это сохраняем в БД).

### Возвраты (refund)

- Инициируются из админки бариста (кнопка «Возврат»).
- Вызов API провайдера на refund; после успеха переводим заказ в `refunded` и логируем в `OrderEvent`.

### План Б: фискализация через Эвотор «Цифровая касса» (готовность)

Если по каким-то причинам фискализация «через Точку и партнёра» станет неудобной/невозможной, фискализацию можно вынести в отдельный модуль:

- Точка подтверждает оплату (`APPROVED`) → заказ становится `paid`.
- После этого наш бэкенд отправляет данные чека в **Эвотор «Цифровая касса»** (публично заявлена совместимость с API АТОЛ Онлайн) и отдельно отслеживает результат формирования чека (poll статуса).

## 2. Печать

### Модель: локальный print-agent (принято)

- В точке (на том же Wi‑Fi или с доступом в сеть) работает **небольшая служба (print-agent)**.
- Бэкенд при переходе заказа в `ready` создаёт **PrintJob** (номер заказа, состав, время) и отправляет его в очередь. Агент периодически опрашивает API или получает задания по HTTPS и печатает на термопринтере по **ESC/POS** (USB или сеть).
- **Плюсы**: не зависит от того, закрыл ли бариста вкладку браузера; можно ретраи и повторную печать из админки; один агент может обслуживать несколько принтеров.
- **Минусы (важно для тестирования и внедрения)**: в каждой точке нужно **установить и настроить** агент — отдельная программа/служба на компьютере или устройстве рядом с принтером. Без агента печать из облачного сервиса не заработает. Нужно заранее описать: как установить (установщик, Docker или скрипт), как зарегистрировать принтер в системе, как проверить печать (тестовый чек), что делать при сбое (логи, «Печать ещё раз» в админке). Для тестирования полезно иметь режим «без реального принтера»: агент помечает задание выполненным и пишет чек в файл или в лог — чтобы гонять сценарии без железа.

### Формат чека (для этапа 5)

- Крупно: **номер заказа** (3–4 цифры).
- Строкой: время готовности.
- Опционально: короткий состав (названия позиций без полной кастомизации для экономии бумаги).
- Мелочью: **QR или штрихкод номера** на чеке — на будущее (удобно, если позже будет считывание с телефона). Если пойдём через ЮKassa, у них есть поддержка QR на чеках — можно будет заложить печать QR на наш чек с номером заказа.

### Протокол доставки заданий (уточнить в Этапе 5)

**В чём разница:**

- **Polling (опрос)**: агент сам раз в N секунд спрашивает у сервера «есть новые задания?», забирает их и печатает. Не нужен белый IP и открытый порт в точке; проще настроить, но небольшая задержка (до N секунд).
- **Push (доставка)**: сервер сразу шлёт задание на известный адрес агента («вот новый чек — напечатай»). Мгновеннее, но агент должен быть доступен из интернета (или через туннель), что сложнее в настройке и безопасности.

**Гибрид**: по умолчанию — polling (просто и надёжно). Если у точки есть возможность принимать входящие запросы (например, агент за туннелем или в облаке), можно дополнительно слать push при появлении задания, а агент при опросе всё равно подхватывает то, что не получил по push. Так и задержка меньше, и при сбое push задание не теряется.
