# Точка.API: платёжные ссылки (и чек `With Receipt`)

Эта страница — практическая БЗ для интеграции оплаты в Coffee Stop Soft через **Точка.API “Платёжные ссылки”**.

## Роли (чтобы не путаться)

- **Точка**: принимает деньги (эквайринг) и даёт **страницу оплаты** по ссылке.
- **Фискализация “через Точку”**: при создании ссылки **с чеком** (`With Receipt`) мы передаём состав заказа и контакты — дальше чек фискализируется партнёром, выбранным в Точке, по требованиям 54‑ФЗ.
- **Эвотор “Цифровая касса”** (план Б): отдельный фискальный контур, если надо фискализировать не через партнёра Точки.

## Включение и подготовка на стороне Точки

1) В интернет-банке Точки подключить **Интернет-эквайринг**:\n
«Сервисы → Интернет-эквайринг».

2) Убедиться, что торговая точка активна (через `Get Retailers`):\n
должно быть `status=REG` и `isActive="true"`.

3) Если используем фискализацию через партнёра Точки — выбрать партнёра при подключении интернет-эквайринга.

## Модель в нашем сервисе (как это ложится на Order/Payment)

Наша модель остаётся простой:

- `draft` → пользователь нажал «Оплатить»
- `payment_pending` → мы создали платёжную ссылку
- `paid` → только когда пришёл webhook/подтверждён статус операции
- дальше производственный поток (`ready`, уведомление, печать и т. д.)

## Создание платёжной ссылки (checkout)

### Минимальный набор параметров (без чека)

При вызове `Create Payment Operation` передаём:

- `amount` — сумма **в рублях**
- `customerCode` — наш код клиента в Точке (из справочников Точки)
- `purpose` — назначение платежа (видно покупателю)
- `paymentMode` — способы оплаты (`card`, `sbp`, `tinkoff`, `dolyame`; допускается несколько)

Рекомендуем передавать дополнительно:

- `paymentLinkId` — **наш** стабильный идентификатор заказа (например, `order_id`)
- `redirectUrl` / `failRedirectUrl` — возврат гостя на наши страницы статуса
- `merchantId` — если у нас несколько торговых точек у Точки
- `ttl` — срок жизни ссылки (в минутах)

### Создание ссылки с фискализацией (наш режим)

Используем `Create Payment Operation With Receipt`, где добавляем:

- `client.email` и/или `client.phone` — чтобы отправить электронный чек гостю
- `items[]` — позиции заказа:\n
  - `name`\n
  - `quantity`\n
  - `amount` (цена за единицу)\n
  - `measure` (обычно штуки)\n
  - `paymentObject` (goods/service/work)\n
  - `paymentMethod` (например `full_payment` или `full_prepayment` — выбрать и зафиксировать в настройках)\n
  - `vatType` (ставка НДС)\n
- `taxSystemCode` — система налогообложения

Важно: в этом режиме **фискализация завязана на состав заказа**. Поэтому сумма и позиции должны вычисляться и валидироваться сервером, а не фронтом.

## Webhook и подтверждение оплаты

### Принцип

- Редирект гостя на `redirectUrl` — **не** источник истины.\n
  Истина — webhook или подтверждённый статус операции.

### Событие

Для платёжных ссылок Точка рекомендует webhook с событием **`acquiringInternetPayment`** (см. статью про платёжные ссылки).

Успех, который переводит заказ в `paid`: **`status="APPROVED"`**.

### Идемпотентность webhook

Webhook может прийти повторно. Наш обработчик обязан:

- дедуплицировать по `operationId`\n
- выполнять переход `payment_pending → paid` один раз\n
- не назначать второй раз `public_number`\n
- не отправлять повторно уведомления/письма

## Refund (возврат)

Возврат запускается из бариста-админки и возможен для операций в статусе `APPROVED`.

Мы должны хранить `operationId`, потому что refund делается именно по нему.

## Что сохраняем в БД (минимальный набор)

- `payment_provider`: `tochka_payment_links`
- `payment_link_id`: то, что мы передали как `paymentLinkId` (обычно `order_id`)
- `operation_id`: то, что пришло в webhook (и/или можно получить из API списка операций)
- `provider_payload`: сырой JSON ответа/вебхука (для дебага)
- `status`: `payment_pending` → `succeeded`/`failed`

## Тестирование

Минимальный тест-план:

- Создание `draft` заказа → `checkout` → получаем `payment_url`.\n
- Открываем `payment_url`, проводим оплату.\n
- Проверяем, что редирект вернул на статус-страницу, но `paid` выставился только после webhook.\n
- Проверяем идемпотентность:\n
  - повторный `checkout` возвращает тот же `payment_url`\n
  - повтор webhook не меняет `public_number` и не создаёт дубликаты событий\n
- Проверяем `refund` из админки: статус заказа/платежа и отражение в аудите (`OrderEvent`).
