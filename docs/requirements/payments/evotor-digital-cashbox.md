# Эвотор «Цифровая касса»: фискализация (план Б)

Эта страница описывает запасной вариант фискализации чеков **не через партнёра Точки**, а напрямую через **Эвотор «Цифровая касса»**.

## Зачем это нужно, если деньги принимает Точка

- **Точка** — принимает оплату (деньги).
- **Фискализация** — формирование фискального чека и передача данных в ОФД/ФНС по 54‑ФЗ.

Если мы используем `With Receipt` в Точке, то фискализация уходит в контур партнёра Точки.\n
Если же нужно контролировать/заменить фискализацию, можно интегрироваться с отдельным сервисом — например, Эвотор «Цифровая касса».

## Что публично известно из документации Эвотор

- «Цифровая касса» — **облачная касса** для автоматизированной фискализации дистанционных оплат.
- Есть **открытое API**.
- API совместим с **АТОЛ Онлайн** (в большинстве сценариев «достаточно заменить URL и креды»).
- Есть тестовый контур.

Официальные ссылки:

- База знаний / быстрое тестирование: `https://docs.evotor.cloud/docs/api/bystraya-proverka-integracii/`
- Реквизиты тестового стенда: `https://docs.evotor.cloud/docs/api/rekvizity-testovogo-stenda`
- Описание ПО «Цифровая касса» (функциональные характеристики): `https://support.evotor.ru/api/article/file/3404`
- Для понимания протокола ATOL: `https://help.atol.online/article/11244`

## Тестовый контур (из Эвотор документации)

В документации тестового стенда указано:

- `URL = https://fiscalization-test.evotor.ru`
- `login = partnerTest1.2`
- `pass = partnerTest1.2Pass`
- `group_code = cf0a2212-2e39-463f-9bff-30874b570f75`

Эти данные нужны только для тестовой проверки интеграции.

## Как это ложится на наш order flow

Поток остаётся прежним:

1) Гость оплачивает через Точку → webhook `APPROVED` → заказ становится `paid`.
2) **После** этого запускаем фискализацию:\n
   наш бэкенд формирует фискальный чек по данным заказа и отправляет в Эвотор.
3) Отслеживаем статус фискализации (poll статуса чека).\n
4) Логируем результат (успех/ошибка), сохраняем идентификаторы.

Важно: **производство не должно блокироваться** на фискализации.\n
Заказ уже `paid`, бариста может готовить. Но фискализация должна быть наблюдаемой (ретраи, алерты).

## Какие данные нужны для фискального чека

Минимальный набор для чека прихода (sell receipt):

- Идентификатор заказа (`order_id`)\n
- Список позиций:\n
  - `name`\n
  - `quantity`\n
  - `price`/`amount`\n
  - ставка НДС (если применимо)\n
  - признаки предмета/способа расчёта (goods/service, prepayment/full_payment)\n
- Сумма\n
- Контакты покупателя для электронного чека (email/phone)\n
- Налоговая система (`taxSystemCode`/эквивалент в протоколе)\n
- Данные организации/ККТ/группы (в терминах Эвотор/АТОЛ)

## Проектные требования к нашей БД и коду (чтобы план Б был простым)

Хранить в `payments` (или отдельной сущности `fiscal_receipts`):

- `fiscal_provider` (например `evotor_digital_cashbox`)\n
- `fiscal_status` (`pending`/`done`/`failed`)\n
- `fiscal_uuid` (идентификатор чека)\n
- `receipt_payload` (JSON, который отправили)\n
- `fiscal_result_payload` (JSON ответа/статуса)\n
- `attempts`, `last_error`, timestamps

## Ошибки и ретраи

- Сетевые ошибки/5xx: ретраи с backoff.\n
- Ошибки валидации чека: перевод в `failed`, алерт в админке/логах.\n
- Дубликаты: дедуплицировать по `order_id` + тип чека (приход/возврат) и не отправлять повторно без явного “retry”.

## Возвраты и фискализация возврата

Если делаем refund денег через банк, то параллельно может требоваться **фискальный чек возврата прихода**.\n
Поэтому дизайн должен предусматривать два независимых шага:

- банковский refund (Точка)\n
- фискализация возврата (Эвотор/АТОЛ-протокол)

## Чего в этой БЗ специально нет

Эта страница не даёт конкретных JSON-схем и URL эндпоинтов «sell/report», потому что:\n
публичные материалы Эвотор подчёркивают совместимость с АТОЛ Онлайн, а точные протоколы/эндпоинты зависят от версии API (v4/v5) и выдаются в техдоках/личном кабинете.\n
Мы фиксируем архитектуру и требования к нашему коду/БД, чтобы интеграция была прямолинейной, когда появятся точные протоколы.
