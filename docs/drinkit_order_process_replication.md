# Процесс обработки заказов в кофейне (по мотивам Дринкит)

Детальное описание процесса «от заказа до выдачи», собранное из открытых источников (сайт Дринкит, vc.ru, Dodo Brands, статьи про приложение). Цель — дать основу для повторения логики в своём проекте.

---

## 1. Общая схема

```
Клиент                    Система                      Точка
   │                         │                            │
   │  Заказ (приложение/      │                            │
   │  киоск) + оплата         │  Заказ в Dodo IS           │
   │─────────────────────────>│───────────────────────────>│ Планшет бариста
   │                         │                            │
   │                         │  Статусы (принят →         │
   │  Уведомления о статусе   │  в работе → готов)         │
   │<─────────────────────────│<───────────────────────────│
   │                         │                            │
   │  Приход в точку         │                            │
   │  Идентификация (QR /     │  Открытие ячейки           │
   │  номер + имя на ячейке)  │                            │
   │─────────────────────────>│───────────────────────────>│ Умная выдача
   │                         │                            │
   │  Забрал заказ           │                            │
   │<─────────────────────────────────────────────────────│
```

## 2. Каналы приёма заказа

| Канал        | Описание |
|-------------|----------|
| **Мобильное приложение** | Основной. Клиент заказывает по пути (из дома, офиса, с улицы). Доля заказов через приложение в Дринкит — 42–66% в зависимости от типа точки. |
| **Киоск в кофейне**      | iPad/планшет на стойке. Для тех, кто без приложения. Без кассира — только экран заказа. |

Важно: **касса в телефоне гостя** — оплата и оформление происходят до прихода в точку, чтобы убрать очередь у кассы.

## 3. Пошаговый сценарий (клиент)

### 3.1 Регистрация и вход

- Установка приложения (или одноразовое использование киоска без аккаунта, если предусмотрено).
- Авторизация (телефон/email и т.п.).
- Защита от фрода при акциях: привязка к устройству/аккаунту, лимиты (опыт Дринкит с акцией «3 напитка по 1 ₽»).

### 3.2 Выбор точки

- Карта или список кофеен.
- Выбор конкретной точки выдачи (где клиент планирует забрать заказ).

### 3.3 Формирование заказа

- **Меню**: напитки, еда (тосты, каши, десерты и т.д.).
- **Кастомизация напитка** (критично для повторения):
  - тип молока (в т.ч. растительное);
  - сиропы (название, количество «порций»);
  - крепость (доп. шот эспрессо);
  - температура (горячий/холодный/лёд);
  - добавки (посыпка, сливочная пенка и т.д.);
  - опции вроде «без сахара».
- Желательно: **«как обычно»** — подсказка последнего/частого заказа по пользователю.
- В Дринкит кастомизируют ~45% заказов; калории считаются по ходу настройки.

### 3.4 Время получения (опционально)

- В части описаний упоминается **время/слот получения** («pick up at your designated time slot»).
- В других — просто «забрать через несколько минут». Для репликации разумно: либо выбор «через N минут» / слот времени, либо только уведомление «готово».

### 3.5 Оплата

- Оплата в приложении (карта, СБП, и т.д.) до прихода в точку.
- Чек/подтверждение заказа (номер заказа, состав, точка, при необходимости время).

### 3.6 Ожидание и уведомления

- Статусы заказа в приложении и push:
  - принят;
  - готовится;
  - **готов к выдаче** — клиент идёт к стойке.
- Оценка: время от заказа до «готов» — порядка нескольких минут (зависит от загрузки точки).

### 3.7 Получение в точке

- Клиент приходит к **умной выдаче** (стойка с ячейками).
- **Идентификация заказа** (по материалам Dodo Brands и vc.ru):
  - на ячейке отображаются **номер заказа и имя гостя**;
  - выдача/разблокировка ячейки — по **QR-коду** из приложения (подносит телефон к считывателю).
- Клиент забирает заказ, ячейка закрывается. Дальше — оценка и повторный заказ в приложении при желании.

## 4. Пошаговый сценарий (точка / бэкенд)

### 4.1 Приём заказа

- Заказ из приложения или киоска уходит в **единую систему** (в Дринкит — Dodo IS).
- Сразу:
  - фиксация состава, кастомизации, точки выдачи;
  - привязка к пользователю и номер заказа;
  - генерация QR для выдачи (привязанный к заказу и ячейке при размещении).

### 4.2 На кухне/баре

- Заказ приходит на **планшет бариста** (кухонный дисплей).
- Бариста принимает заказ в работу, статус «в работе».
- Приготовление по рецепту (напиток + при необходимости еда).
- Готовый заказ кладётся в **конкретную ячейку** умной выдачи.
- В системе: «заказ в ячейке N», статус «готов к выдаче».
- Клиенту уходит push «заказ готов».

### 4.3 Умная выдача

- Стойка с ячейками (lockers).
- На каждой ячейке — экран/подсветка: **номер заказа + имя** (из Dodo Brands).
- Разблокировка по сканированию **QR из приложения** (или по коду из СМС/приложения, если без QR).
- Логика: один заказ — одна ячейка; при получении ячейка разблокируется, затем снова доступна для следующих заказов.
- Оборудование: разработка в Dodo — через «лабораторию IoT» (сборка и тесты своих устройств).

### 4.4 Аналитика и операции (как в Dodo IS)

- Учёт сырья по рецептам, списание при заказе.
- Прогноз спроса, подсказки по загрузке персонала по часам.
- Отчёты: продажи, популярные позиции, время приготовления, повторные заказы.

## 5. Что нужно для репликации

### 5.1 Клиентская часть

| Компонент | Назначение |
|-----------|------------|
| Мобильное приложение (или PWA) | Меню, кастомизация, корзина, оплата, статусы, QR для выдачи. |
| Киоск (веб или нативное приложение на планшете) | Тот же поток заказа + выбор точки = эта же кофейня. Без кассы. |

### 5.2 Бэкенд

| Компонент | Назначение |
|-----------|------------|
| API приёма заказов | Принять заказ из приложения/киоска, сохранить, выдать номер и QR. |
| Управление статусами | Принят → В работе → Готов; уведомления в приложение (push/поллинг). |
| Интеграция с оплатой | Приём платежа до фиксации заказа (СБП, эквайринг и т.д.). |
| Привязка заказа к ячейке | Когда бариста кладёт заказ в ячейку — в системе фиксируется «заказ X → ячейка N». |

### 5.3 В точке

| Компонент | Назначение |
|-----------|------------|
| Планшет/дисплей для бариста | Очередь заказов, смена статуса (принят → в работе → готов), привязка к ячейке. |
| Умная выдача | Ячейки с экраном (номер + имя), замок, считывание QR (или ввод кода). Связь с бэкендом: «открыть ячейку для заказа X по коду/QR». |
| Оборудование ячеек | Контроллеры, замки, сканер QR; прошивка и API для открытия по команде с сервера. |

### 5.4 Без умной выдачи (упрощённый вариант)

Если пока не делать ячейки с QR:

- Выдача «по номеру и имени»: на экране/табло в зале список готовых заказов (номер + имя); клиент называет номер, сотрудник отдаёт заказ из стойки или с полки.
- Либо клиент показывает в приложении экран с номером заказа — сотрудник сверяет и отдаёт.

Потом этот же поток можно доработать до полноценной умной выдачи с ячейками и QR.
